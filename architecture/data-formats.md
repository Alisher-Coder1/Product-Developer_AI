# Data Formats — Core MVP v1.1

Документ фиксирует утверждённые форматы обмена данными в Product Engineering OS, требования к их оформлению, валидации и версионированию.

---

## 1. Утверждённые форматы

### JSON

Используется для:

- конфигураций
- машинного обмена данными
- результатов работы движков
- структурированных ответов CLI

Примеры:

- `architecture/stages-config.json` **(обязательный)**
- результаты Stage Engine (структура описана в спецификации движка)
- результаты AI Task Engine (структура описана в спецификации движка)
- результаты Validation Engine (структура описана в спецификации движка)

Причина выбора:

- простой машинный парсинг
- стандартный формат для JavaScript/Node.js
- удобен для CLI и внутренних модулей

---

### YAML

Используется для:

- API контрактов
- описания схем
- CI конфигураций
- декларативных файлов интеграции

Примеры:

- OpenAPI спецификации (например, `contracts/api/openapi.yaml`)
- GitHub Actions workflows (`.github/workflows/*.yml`)
- внешние contract files

Причина выбора:

- удобочитаемость
- распространённость в DevOps и API tooling
- подходит для декларативных конфигураций

---

### Markdown

Используется для:

- архитектурной документации
- спецификаций движков
- roadmap
- product context
- generated context

Примеры:

- `architecture/engine-specifications.md`
- `architecture/engines/*.md`
- `generated/project-context.md`
- `product/*.md` **(обязательные: vision.md, roadmap.md)**

Причина выбора:

- удобство для людей
- поддержка GitHub
- совместимость с AI context pipeline

## 2. Общие требования к оформлению

### Кодировка
Все файлы должны быть в кодировке **UTF-8**.

### Форматирование

- **JSON**: отступы — 2 пробела, без завершающих запятых, файл должен валидироваться по соответствующей схеме (см. раздел 3).
- **YAML**: отступы — 2 пробела, использование стандарта [YAML 1.2](https://yaml.org/spec/1.2/spec.html), файл должен проходить синтаксическую валидацию.
- **Markdown**: следование спецификации [CommonMark](https://commonmark.org/), переносы строк — LF.

---

## 3. Требования к схемам и валидации

Для всех ключевых конфигурационных JSON-файлов должны быть определены **JSON Schema** (версии **draft-07** или новее), которые описывают допустимую структуру, типы полей и ограничения. Схемы размещаются в папке `schemas/` с именами, соответствующими целевым файлам, например:

- `schemas/stages-config-schema.json` — для `architecture/stages-config.json`
- `schemas/engine-results-schema.json` — для общих результатов движков (опционально)

Сами схемы должны проходить валидацию (например, с помощью AJV) перед использованием.

В рамках Core MVP обязательной валидации подлежат:

- `architecture/stages-config.json`
- любые файлы, генерируемые движками (проверка осуществляется самим движком согласно внутренней логике).

Для YAML-файлов, используемых как контракты (OpenAPI), рекомендуется использовать соответствующие валидаторы (например, `openapi-cli validate`).

---

## 4. Структура результатов движков

Каждый движок (Stage, AI Task, Context Generation, Token Budget, Validation) возвращает результаты в JSON-формате. Точная структура этих результатов **описана в спецификации соответствующего движка** (файлы `architecture/engines/*-spec.md`). При разработке необходимо строго придерживаться этих спецификаций для обеспечения совместимости.

Если результаты движка предназначены для долговременного хранения или обмена между компонентами, они **должны** включать поле `formatVersion` (аналогично конфигурационным файлам), чтобы обеспечить возможность будущих изменений формата.

---

## 5. Версионирование форматов

Для обеспечения обратной совместимости и контролируемых изменений вводится политика версионирования:

- Все конфигурационные JSON-файлы **должны** содержать корневое поле `"formatVersion"` со строковым значением в формате SemVer (например, `"1.0.0"`).
- Изменения, не нарушающие обратную совместимость (добавление полей), увеличивают **minor**-версию.
- Изменения, нарушающие обратную совместимость (удаление или изменение типа полей), требуют увеличения **major**-версии и обновления всех потребителей.
- В рамках Core MVP v1.1 поддерживается только версия `1.x.x`.

Пример:
```json
{
  "formatVersion": "1.0.0",
  "stages": {
    ...
  }
}
6. Примеры
Пример YAML-файла (фрагмент OpenAPI спецификации)
Ниже приведён фрагмент OpenAPI описания. Полный файл должен соответствовать спецификации OpenAPI 3.0 и может быть проверен утилитами вроде openapi-cli.

yaml
openapi: 3.0.0
info:
  title: Product API
  version: 1.0.0
paths:
  /products:
    get:
      summary: Список продуктов
      responses:
        '200':
          description: Успешный ответ
          content:
            application/json:
              schema:
                type: array
                items:
                  $ref: '#/components/schemas/Product'

8. Ограничения Core MVP v1.1
В рамках Core MVP v1.1 утверждены только следующие форматы:

JSON

YAML

Markdown

Другие форматы не входят в базовый стандарт этапа 0 и могут быть добавлены только после согласования архитектором.

Итог
Форматы обмена данными для Core MVP v1.1 утверждены, дополнены требованиями к оформлению, валидации и версионированию. Все разработчики обязаны соблюдать данный документ при создании и модификации файлов в проекте.
