Product Engineering OS — Architecture Freeze v1.1

Версия v1.1 собрана после внешнего аудита и фиксирует обновлённую архитектурную позицию платформы: лёгкое, но масштабируемое ядро для сложных задач. Документ предназначен для повторного архитектурного аудита и служит базой для последующего Roadmap как SSOT.

Ключевое изменение версии 1.1: переход от концепции «полной платформы сразу» к модели Core MVP + Advanced Extensions.

АРХИТЕКТУРНОЕ РЕШЕНИЕ

1. Базовая позиция системы

Product Engineering OS — это AI-native engineering framework для одного разработчика или небольшой команды.

Цель платформы:

  • быстро запускать продукты со сложной логикой;
  • сохранять архитектурную дисциплину при работе с AI;
  • уменьшать галлюцинации модели через управляемый контекст;
  • обеспечивать пошаговую разработку через этапы, задачи и микрозадачи;
  • оставлять ядро простым, а сложность подключать модулями.
Формула платформы: Simple Core + Stable Interfaces + Optional Modules = Scalable Platform.

2. Главные архитектурные принципы

Documentation Driven Development

Ключевые знания системы фиксируются в документах, а не в неявных договорённостях.

Domain Driven Design

Домен определяет бизнес-понятия, правила и инварианты.

Contract First Architecture

Интерфейсы системы фиксируются до реализации.

Block Based Implementation

Код реализуется блоками с чёткими границами ответственности.

AI Context Architecture

AI получает только релевантный и слоисто организованный контекст.

Microtask Driven Development

Большие задачи дробятся на маленькие проверяемые шаги.

Stage Gating

Нижние этапы не запускаются, пока верхние артефакты не зафиксированы.

Core First

Сначала реализуется MVP ядра, затем advanced-расширения.

MVP CORE

3. Архитектурное решение по составу ядра

После аудита принято решение строить сначала Core MVP с минимальным набором движков.

EngineНазначение
Stage EngineУправляет последовательностью этапов и stage gates.
AI Task EngineПревращает задачи в микрозадачи и управляет их исполнением.
Context GenerationСобирает актуальный проектный контекст для AI.
Token BudgetСчитает и ограничивает расходы токенов до запуска prompt.
ValidationПроверяет результат AI через tests, lint, contracts и базовые quality gates.
Всё остальное — Context Router, Vector Knowledge, Reuse Engine, Design Source Engine, Full AI Pipeline — выносится в advanced-расширения и не является обязательной частью v1 ядра.

3.1 Спецификации движков (кратко)

Каждый движок описывается в architecture/engines/*-spec.md с указанием:

  • форматы входных/выходных данных (JSON, YAML, Markdown);
  • алгоритм работы (псевдокод или диаграмма);
  • пример использования;
  • интерфейсы взаимодействия с другими движками.

Пример сигнатуры AI Task Engine:

Вход: { task: "Добавить аутентификацию", layer: "application", constraints: [...] }
Выход: [ { microtask: "Создать модель User", ... }, ... ]

5. Dependency Rules

Зависимости разрешены только сверху вниз по утверждённой иерархии.

product → architecture → domain → application → contracts → blocks
  • domain не зависит от application, contracts, blocks;
  • application зависит только от domain и architecture rules;
  • contracts фиксируют стабильные интерфейсы и не зависят от реализации;
  • blocks реализуют contracts и use cases, но не меняют domain semantics;
  • quality / security / operations применяются поперечно к реализации и валидации.

Пример автоматической проверки: npx depcruise --validate .dependency-rules.js src (используется dependency-cruiser).

9. AI Knowledge Policy

  • общие знания программирования — не индексируются;
  • project-specific intelligence — индексируются: архитектура, домен, контракты, решения, стандарты кода.

11. Token Budget Policy

Prompt TypeRecommended Budget
Product / Architectureдо 12k
Domain / Applicationдо 8k
Implementation microtaskдо 10k
Review / Fixдо 6k

Если лимит превышен: использовать summary mode, уменьшить контекст, предложить разбить задачу.

Пример работы Token Budget

Запрос: реализовать REST API для пользователей. Контекст содержит 15k токенов. Действие: включается summary mode – ключевые части контекста (схема БД, основные эндпоинты) сохраняются, второстепенные детали обрезаются. Итоговый промпт: 9.8k токенов.

12. Validation & Testing

Validation Engine связан с тестированием и использует:

  • unit tests — Jest (JavaScript/TypeScript), PyTest (Python);
  • integration tests — Supertest, TestContainers;
  • contract tests — Pact, OpenAPI validator;
  • architecture compliance — Dependency cruiser, ESLint (import rules);
  • basic security — Trivy (зависимости), ESLint-plugin-security.

Validation loop:

AI output
↓
lint / tests / contract checks / architecture checks
↓
pass → accept
fail → fix prompt → re-run

13. Feedback Loop

Если наблюдаются:

  • повторные ошибки AI на одном типе задач;
  • постоянные нарушения контрактов;
  • двусмысленность в use cases или domain rules;
  • частые проблемы в validation loop;

то это сигнал обновить архитектурные документы, code standards, task specifications, prompt templates.

14. Architecture Compliance

Минимальные compliance checks:

  • нарушение layer dependencies;
  • нарушение naming / code standards;
  • расхождение contracts и реализации;
  • отсутствие обязательных tests;
  • нарушение prompt data safety.

Пример нарушения: код из backend обращается к фронтенд-компоненту — блокируется CI.

19. CLI as Mandatory Platform Entry Point

Основной способ работы — через командный интерфейс pos.

Обязательные команды Core MVP

pos init          # создание структуры проекта
pos start         # запуск guided workflow
pos context       # сборка project-context.md
pos prompt        # сборка промпта для задачи
pos tokens        # оценка токенов и контроль бюджета
pos validate      # запуск validation loop

CLI реализуется в scripts/pos.js с модулями generate-context, assemble-prompt, token-count, validate-architecture.

21. Примеры артефактов

Пример project-context.md (фрагмент)

# Project Context: MyProduct (2026-03-12) ## System Architecture - Стек: React, Node.js, PostgreSQL - Основные модули: Auth, Billing, Notifications ## Domain Model - User: id, email, role - Order: id, userId, total, status ## Use Case: Создание заказа - Вход: userId, товары - Выход: orderId - Правила: корзина не пуста ## Contracts - POST /api/orders { userId, items[] } → { orderId }

Пример промпта для микрозадачи

Задача: реализовать POST /api/orders в контроллере. Контекст: (ссылка на project-context.md, стандарты кода) Требования: - использовать Express - валидировать вход через Joi - вызвать сервис createOrder - вернуть 201 с orderId

Пример работы Token Budget при превышении

Собран контекст: 18k токенов (лимит 12k). Применён summary mode: удалены детальные логи, оставлены только схемы и ключевые описания. Результат: 11.2k токенов → промпт выполнен.

22. Безопасность промптов (Prompt Data Safety)

В security/prompt-data-safety.md фиксируются правила:

  • автоматическое сканирование промптов на наличие секретов (API-ключи, пароли, токены) с помощью регулярных выражений и детекторов (например, detect-secrets);
  • маскирование чувствительных данных: замена на [REDACTED] перед отправкой в AI;
  • логирование промптов только в обезличенном виде (удаление персональных данных);
  • запрет на включение в контекст файлов с секретами (.env, *secret*).

23. Quick Start (как начать использовать)

# Установка (через npm)
npm install -g @product-os/cli

# Создание нового проекта
pos init my-project
cd my-project

# Запустить guided flow для первой фичи
pos start

# Собрать контекст вручную
pos context

# Оценить токены для текущей задачи
pos tokens --task "описание"

# Выполнить валидацию после генерации
pos validate
